# hi-im-hubclient 技术设计文档

> **组件**：hi-im-hubclient（Go module）  
> **层级**：L2 Hub TCP 客户端（**不独立部署**）  
> **必嗨等价**：`lib/rtmq`（Go Proxy 部分）  
> **版本**：v0.1 · 2026-07-02  
> **生态**：对应 [hi-im/doc/hi-im-档C技术方案设计.md](https://github.com/sunchao1/hi-im/blob/main/doc/hi-im-档C技术方案设计.md) **生态 M2**

---

## 1. 定位与边界

### 1.1 是什么

**hi-im-hubclient** 是 Go 业务进程内嵌的 **Hub 客户端库**，负责：

- 与 **hi-im-core**（`hi-im-hub`）建立 **TCP** 长连接（FORWARD 或 BACKEND 平面）
- 完成 **AUTH / SUB / KPALIVE** 系统握手
- 向 Hub **上行**业务帧（`AsyncSend`）
- 接收 Hub **下行**业务帧，按 **cmd** 分发给注册的 `Handler`
- 断线 **自动重连**，重连后重新 AUTH + SUB

对应必嗨文献中的 **RTMQ Proxy**——是 **Hub 的客户端**，不是 GoF Proxy 模式，也不是对外 **hi-im-sdk-go**。

### 1.2 不是什么

| 不是 | 说明 |
|------|------|
| **独立部署的服务** | 无 `main`、无 Pod、无 HTTP 端口 |
| **hi-im-api** | api 定义 IM 48B 头与 CMD；hubclient 只搬运 **opaque payload** |
| **业务逻辑** | 无群 fan-out、无 Redis、无 WebSocket、无 Gin |
| **CGO 绑定 hi-im-core** | 纯 Go 实现 bus wire v1；运行时 TCP 连 C++ Hub |
| **gRPC 客户端** | 冷路径 RPC 在 L3 服务内直接调 seqsvr |

### 1.3 与 hi-im-api 分工

```text
msgsvr 群聊下行示例:
  1. imv1.GroupChat 序列化              ← hi-im-api (proto)
  2. header.Pack(cmd, sid, destNid, …)  ← hi-im-api (IM 头)
  3. hubclient.AsyncSend(cmd, destNid, frame) ← hi-im-hubclient (20B wire + payload)
  4. Hub bridge 读 IM 头 / wire.nid → async_send 到 gateway
```

| | **hi-im-api** | **hi-im-hubclient** |
|--|---------------|---------------------|
| 层级 | 契约 | 传输 |
| 变更频率 | 新 cmd / proto 时常改 | **很少**（wire v2、多 shard 时才改） |
| 依赖方向 | 无 hubclient 依赖 | `go get` hi-im-api（**仅测试与示例**；库本身不 import 业务 proto） |

**原则**：hubclient **保持很薄**。若每次加业务功能都要改 hubclient，说明业务渗入了传输层（见档 C §4.6）。

## 2. 在协议栈中的位置

```text
┌──────────────────────────────────────────────────────────────┐
│ L3  hi-im-gateway / msgsvr / chatroom / usrsvr / roomfanout   │
│      import hubclient + hi-im-api                             │
└────────────────────────────┬─────────────────────────────────┘
                             │ hubclient.Client
┌────────────────────────────▼─────────────────────────────────┐
│ L2  hi-im-hubclient（本仓库）                                   │
│      TCP · AUTH/SUB/KPALIVE · AsyncSend · RegisterHandler     │
│      [ WireHeader 20B | payload ]                               │
└────────────────────────────┬─────────────────────────────────┘
                             │ bus wire v1
┌────────────────────────────▼─────────────────────────────────┐
│ L1  hi-im-core hi-im-hub                                      │
│      FORWARD :28888  ·  BACKEND :28889  ·  bridge             │
└──────────────────────────────────────────────────────────────┘

payload 典型内容（由 L3 + hi-im-api 构造，Hub 视大部分为 opaque）:
  [ IM MesgHeader 48B/52B | protobuf body ]
```

**两层头**（实现时勿混）：

| 层 | 长度 | 定义仓库 | hubclient 职责 |
|----|------|----------|----------------|
| **bus wire** | 20B | hi-im-core 规范 | **编解码、校验 chksum** |
| **IM MesgHeader** | 48B（文档简称；beehive wire 52B） | hi-im-api | **不解析**（payload 透传） |

线协议权威文档：[hi-im-core/doc/协议规范-bus-wire-v1.md](https://github.com/sunchao1/hi-im-core/blob/main/doc/协议规范-bus-wire-v1.md)。

## 3. 必嗨迁移对照

| 必嗨 | hi-im-hubclient |
|------|-----------------|
| `lib/rtmq/rtmq_proxy.go` | `pkg/hubclient` + `pkg/hubclient/conn` |
| `ProxyInit` / `Launch` | `New` / `Start` |
| `Register(cmd, cb, param)` | `RegisterHandler(cmd, Handler)` |
| `AsyncSend(cmd, data, len)` | `AsyncSend(cmd, destNid, payload)` |
| `ProxyConf` | `Config` |
| 10 条 sendq/recvq（`RTMQ_MSGQ_NUM`） | `ConnPool`（M2 可先单连接，M2 末或多连接） |
| beego/logs | `log/slog`（可 `WithLogger` 注入） |

### 3.1 API 演进：`destNid` 显式化

必嗨 Go Proxy 的 `AsyncSend` 第一参数名为 `cmd`，但 **msgsvr fan-out** 曾把目的 NID 误传为第一参数；实际下行路由依赖 **IM 头内的 Nid 字段**，wire 头 `nid` 始终为 **本进程 NID**。

hi-im-hubclient **统一语义**：

```go
// destNid == 0：wire.nid = 本进程 NID（gateway 上行、msgsvr publish 到 Hub）
// destNid > 0：wire.nid = destNid（Hub async_send 单播语义；payload 仍为完整 IM 帧）
func (c *Client) AsyncSend(cmd, destNid uint32, payload []byte) error
```

与 [hi-im-core 协议规范 §5](https://github.com/sunchao1/hi-im-core/blob/main/doc/协议规范-bus-wire-v1.md) 一致：

| 方向 | wire.flag | wire.nid | wire.type |
|------|-----------|----------|-----------|
| Proxy 上行 | EXP(1) | **本进程 NID** | 业务 cmd |
| Hub async_send 下行 | EXP(1) | **目的 NID** | 业务 cmd |

### 3.2 相对必嗨的刻意改进

| 项 | 必嗨行为 | hi-im-hubclient |
|----|----------|-----------------|
| AUTH → SUB 顺序 | `auth()` 与 `subscribe()` 连续入队，不等待 ACK | **收到 AUTH_ACK 成功后再 SUB** |
| 退出 | `Stop` 关闭 exitq | `Close` + `context` 取消，WaitGroup 等待 goroutine |
| 日志 | 强依赖 beego | 标准库 `slog`，零强制第三方日志 |
| 测试 | 无单测 | wire golden + integration（对 hi-im-hub） |

## 4. 连接模型

### 4.1 双平面

| 平面 | 默认端口 | 典型进程 | 说明 |
|------|----------|----------|------|
| **FORWARD** | 28888 | hi-im-gateway | 面向接入 NID；接收 Hub async_send 下行 |
| **BACKEND** | 28889 | msgsvr / chatroom / usrsvr / roomfanout | 接收 publish；上行或带 destNid 单播 |

每个 Go 进程通常 **只连一个平面、一个 Hub 地址**（分片 Phase 2 再扩展 multi-backend）。

### 4.2 单连接 vs 连接池

必嗨为每条逻辑连接创建 **最多 10 条 TCP**（`RTMQ_MSGQ_NUM`），`AsyncSend` 随机选一条，用于提高吞吐。

| 阶段 | 策略 |
|------|------|
| **M2** | 默认 **1 条 TCP** 跑通 AUTH/unicast |
| **M2 末 / M3** | 可选 `Config.PoolSize = 10`，行为对齐必嗨 |
| **调优** | 压测后再决定是否默认多连接 |

### 4.3 NID 与鉴权

- 每个 hubclient 实例绑定唯一 **`Config.NID`**，集群内不可冲突。
- **AUTH_REQ** payload（与必嗨 `rtmq_link_auth_req_t` 一致）：

| 字段 | 类型 | 说明 |
|------|------|------|
| gid | uint32 BE | 分组 |
| user | char[32] | 用户名 |
| passwd | char[16] | 密码 |

- wire 头 **`nid`** 字段在 AUTH 帧上仍为 **本进程 NID**（非 payload 内字段）。
- **AUTH_ACK** body：`is_succ uint32`（0 失败，1 成功）。

## 5. 生命周期与状态机

### 5.1 状态

```text
         ┌─────────┐
         │  Idle   │
         └────┬────┘
              │ Start()
              ▼
         ┌─────────┐     dial fail    ┌──────────┐
         │ Dialing │ ───────────────► │ Backoff  │──┐
         └────┬────┘                  └──────────┘  │
              │ success                             │ retry
              ▼                                     │
         ┌─────────┐                                 │
         │Authing  │ ◄───────────────────────────────┘
         └────┬────┘
              │ AUTH_ACK ok
              ▼
         ┌─────────┐
         │Subscribing│  对每个已 RegisterHandler 的 cmd 发 SUB_REQ
         └────┬────┘
              │ SUB 已发送（M2 可不等 SUB_ACK）
              ▼
         ┌─────────┐
         │  Ready  │  AsyncSend / 收业务帧
         └────┬────┘
              │ TCP 错误 / KPALIVE 超时 / Close()
              ▼
         ┌─────────┐
         │ Closed  │
         └─────────┘
```

### 5.2 握手时序

```text
Client                          hi-im-hub
  │──── TCP connect ───────────────►│
  │──── AUTH_REQ (sys) ────────────►│
  │◄─── AUTH_ACK (sys) ────────────│
  │──── SUB_REQ(cmd₁) (sys) ───────►│  每个订阅 cmd 一条
  │──── SUB_REQ(cmd₂) (sys) ───────►│
  │◄─── SUB_ACK (sys, 可选) ───────│
  │                                 │
  │◄══ KPALIVE_REQ/ACK 每 30s ═════►│
  │                                 │
  │──── EXP cmd + payload ─────────►│  AsyncSend（上行）
  │◄─── EXP cmd + payload ──────────│  handler 回调（下行）
```

### 5.3 保活

| 项 | 值 |
|----|-----|
| 间隔 | **30s**（`RTMQ_KPALIVE_INTV`，可配置） |
| 失败判定 | 连续 **3** 次发出 KPALIVE_REQ 无 ACK → 断线重连 |
| 系统帧优先级 | KPALIVE / AUTH / SUB 走 **系统发送队列**，不与业务帧饿死 |

### 5.4 重连

- 指数退避：初始 1s，上限 30s，加随机 jitter。
- 重连成功后：**重新 AUTH → SUB → Ready**。
- 重连期间 `AsyncSend` 返回 `ErrNotReady` 或阻塞等待（**M2 默认返回 error**；可配置 `SendWaitReady`）。
- 可选回调 `OnReconnect` 供 L3 打 metrics。

## 6. 配置

### 6.1 Config 结构

```go
type Config struct {
    // 连接
    Addr     string   // "host:28888" 或 "host:28889"
    NID      uint32   // 本进程唯一 NID
    GID      uint32   // AUTH 分组，默认 10
    User     string   // AUTH 用户名
    Pass     string   // AUTH 密码

    // 订阅（也可仅通过 RegisterHandler 推导）
    Subscribe []uint32

    // 并发与队列
    PoolSize      int // TCP 连接数，默认 1
    WorkerNum     int // 每连接 handler goroutine，默认 4
    SendQueueLen  int // 默认 50000
    RecvQueueLen  int // 默认 50000

    // 保活 / 重连
    KeepaliveInterval time.Duration // 默认 30s
    KeepaliveMaxRetry int             // 默认 3
    ReconnectMaxBackoff time.Duration // 默认 30s

    // AsyncSend 背压
    SendTimeout time.Duration // 入队超时，默认 1s；对齐必嗨
}
```

### 6.2 环境变量（与档 C §7.3 对齐）

| 变量 | 说明 | 示例 |
|------|------|------|
| `HIIM_FORWARD_ADDR` | gateway 用 FORWARD | `hi-im-hub-0:28888` |
| `HIIM_BACKEND_ADDR` | 业务服务用 BACKEND | `hi-im-hub-0:28889` |
| `HIIM_NID` | 本进程 NID | `20001` |
| `HIIM_SHARD_ID` | 分片 ID（Phase 2；M2 仅记录） | `0` |
| `HIIM_AUTH_USER` | AUTH 用户名 | `websocket` |
| `HIIM_AUTH_PASS` | AUTH 密码 | `***` |
| `HIIM_SUB_CMDS` | 逗号分隔订阅 cmd 十六进制 | `0x030B,0x0101` |

`ConfigFromEnv()` 读取上述变量；L3 仍可在代码里 `RegisterHandler` 补充订阅。

### 6.3 各服务典型配置

| 服务 | 平面 | 典型 SUB cmd | AsyncSend 模式 |
|------|------|--------------|----------------|
| **gateway** | FORWARD | 来自 usrsvr/msgsvr 的下行 cmd | `destNid=0` 上行 |
| **msgsvr** | BACKEND | `GROUP-CHAT` 等（**SUB=1**，见 §9） | fan-out 时 `destNid=gatewayNid` |
| **chatroom** | BACKEND | `ROOM-CHAT` | 同上 |
| **usrsvr** | BACKEND | `ONLINE` / 生命周期 | 按需 |
| **roomfanout** | BACKEND | 通常 **不 SUB** | Kafka 消费后 `destNid=nid` |

## 7. 对外 API

### 7.1 类型

```go
package hubclient

// Handler 在 WorkerNum 个 goroutine 之一中同步调用；勿阻塞。
// origNid：wire 头 nid（下行时为 Hub 指定的源/目的 NID，见回调场景）
type Handler func(cmd, origNid uint32, payload []byte)

type Client struct { /* ... */ }

type Option func(*Client)
```

### 7.2 构造与生命周期

```go
func New(cfg *Config, opts ...Option) (*Client, error)
func DefaultConfig() *Config
func ConfigFromEnv() (*Config, error)

func (c *Client) RegisterHandler(cmd uint32, h Handler) error
func (c *Client) Start(ctx context.Context) error
func (c *Client) Close() error

func (c *Client) WaitReady(ctx context.Context) error
func (c *Client) Ready() bool
```

**约定**：

- `RegisterHandler` 必须在 `Start` 之前调用（含 cmd=`0` 默认 handler）。
- `Start` 非阻塞：内部 goroutine 负责 dial 与重连。
- `Close` idempotent，等待 goroutine 退出。

### 7.3 发送

```go
func (c *Client) AsyncSend(cmd, destNid uint32, payload []byte) error
```

| 步骤 | 行为 |
|------|------|
| 组 wire 头 | `flag=EXP`, `type=cmd`, `length=len(payload)`, `chksum=0x1FE23DC4` |
| 设置 nid | `destNid==0 → Config.NID`；否则 `destNid` |
| 入队 | 写入 send queue；超时返回 `ErrSendTimeout` |
| 线程安全 | 多 goroutine 可并发调用 |

**payload 长度**：M2 不硬限；L3 应遵循 IM 最大包长。Hub 侧有队列背压。

### 7.4 选项与观测钩子

```go
func WithLogger(l *slog.Logger) Option
func WithOnReconnect(fn func()) Option
func WithOnStateChange(fn func(State)) Option  // 可选，供 metrics
```

hubclient **不** 内置 Prometheus HTTP；L3 在 `Handler` / hook 中打指标（档 C：可观测主要在 Hub + 业务）。

### 8.2 Goroutine 模型（单 session）

```text
┌─────────────────────────────────────────────────────────┐
│ Client.Start                                             │
│   └── connPool.start                                     │
│         └── per TCP session:                             │
│               ├── recvLoop    → snap 拼帧 → recvq         │
│               ├── sendLoop    ← sendq + sysq (KPALIVE)   │
│               └── worker×N    ← recvq → route → Handler  │
└─────────────────────────────────────────────────────────┘
```

| Goroutine | 职责 |
|-----------|------|
| **recvLoop** | 阻塞读 TCP，`snap` 拆帧，校验 chksum，系统帧本地处理，业务帧入 recvq |
| **sendLoop** | 优先 sysq，再 sendq；写 wire 头 + body |
| **worker** | 从 recvq 取帧，查 handler map，`Handler(cmd, origNid, payload)` |
| **supervisor** | 保活 ticker、重连退避、状态迁移 |

### 8.3 系统消息处理

| cmd | 处理 |
|-----|------|
| `AUTH_ACK` | 成功 → Subscribing；失败 → 断线 |
| `SUB_ACK` | M2 记录日志即可 |
| `KPALIVE_ACK` | 重置失败计数 |
| `KPALIVE_REQ` | 回复 `KPALIVE_ACK`（Hub 主动保活时） |

### 8.4 下行 dispatch

```go
// 查找顺序：精确 cmd → cmd==0 默认 → 丢弃并 slog.Warn
item, ok := c.handlers[cmd]
```

**注意**：`Handler` 内 **不要** 做 fan-out；fan-out 在 msgsvr/chatroom（L3）。

## 9. SUB 策略（L3 配置，hubclient 只执行）

hubclient 只负责发 `SUB_REQ`；**订阅哪些 cmd、几个副本 SUB** 由 L3 架构决定（档 C §7.4）：

| cmd | 建议 SUB 副本数 | 原因 |
|-----|-----------------|------|
| `GROUP-CHAT` | **1** | 避免 publish 广播重复消费 |
| `ROOM-CHAT` | **1** | 同上 |
| `ONLINE` | 1 或按 shard | usrsvr 可单活 |

msgsvr/chatroom 通过 **环境变量 / 配置** 决定 SUB 列表；hubclient **不** 内置「只 SUB 一次」的分布式锁（锁在 K8s 部署层或 Redis 选主）。

## 10. bus wire v1 实现要点

实现必须与 [hi-im-core 协议规范](https://github.com/sunchao1/hi-im-core/blob/main/doc/协议规范-bus-wire-v1.md) **字节级一致**。

### 10.1 WireHeader（20B）

| 偏移 | 字段 | 说明 |
|------|------|------|
| 0 | type | cmd |
| 4 | nid | 见 §7.3 |
| 8 | flag | 0=系统，1=业务 |
| 12 | length | payload 长度 |
| 16 | chksum | 固定 `0x1FE23DC4` |

### 10.2 系统 cmd

| type | 名称 |
|------|------|
| 0x0001 | AUTH_REQ |
| 0x0002 | AUTH_ACK |
| 0x0003 | KPALIVE_REQ |
| 0x0004 | KPALIVE_ACK |
| 0x0005 | SUB_REQ |
| 0x0006 | SUB_ACK |

### 10.3 SUB_REQ body

```text
uint32 BE  sub_cmd   // 要订阅的业务 CMD
```

### 10.4 Snap 拼帧

```go
// 伪代码
for {
    fill read buffer
    for each complete frame where len>=20 && chksum ok && len>=20+length:
        deliver frame
        consume
}
```

单测覆盖：半包、粘包、连续多帧、非法 chksum 丢弃。

## 11. 错误语义

| 错误 | 含义 |
|------|------|
| `ErrNotReady` | 未 AUTH 或重连中 |
| `ErrSendTimeout` | sendq 满且超过 SendTimeout |
| `ErrClosed` | Client 已 Close |
| `ErrAuthFailed` | AUTH_ACK is_succ=0 |

**Handler 内 panic**：worker  recover + slog.Error，不断开 TCP（对齐必嗨 recover 行为）。

## 14. Phase 2 扩展（非 M2 范围）

| 项 | 说明 |

## 15. L3 嵌入示例

### 15.1 gateway（FORWARD）

```go
cfg := hubclient.DefaultConfig()
cfg.Addr = os.Getenv("HIIM_FORWARD_ADDR")
cfg.NID = mustU32(os.Getenv("HIIM_NID"))
cfg.User = os.Getenv("HIIM_AUTH_USER")
cfg.Pass = os.Getenv("HIIM_AUTH_PASS")

cli, _ := hubclient.New(cfg, hubclient.WithLogger(logger))
cli.RegisterHandler(cmd.CMD_GROUP_CHAT, onGroupChatDownlink)
cli.RegisterHandler(0, onUnknownCmd)
_ = cli.Start(ctx)
_ = cli.WaitReady(ctx)

// WS 收到上行帧后（header 已由 hi-im-api 解出）:
_ = cli.AsyncSend(head.Cmd, 0, rawFrame)
```

### 15.2 msgsvr（BACKEND，群聊 fan-out 的一段）

```go
cli.RegisterHandler(cmd.CMD_GROUP_CHAT, onGroupChatUplink)

// fan-out 到 gateway NID=20002（payload 为完整 IM 帧）:
_ = cli.AsyncSend(cmd.CMD_GROUP_CHAT, 20002, frame)
```
